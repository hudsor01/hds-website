<objective>
Resolve ADMINUX-01 by removing the misleading, always-"Admin" `pageTitle` prop from the admin Topbar and its single consumer in the admin layout.

Research finding (HIGH confidence): per-page titles ALREADY exist on all 18 admin routes — each page renders its own `<h1>` (inline or via `ResourceListPage title=`) and exports `metadata.title`. The Topbar's `pageTitle="Admin"` is a redundant, always-wrong duplicate sitting in the chrome above each page's correct heading. In the Next.js App Router a layout structurally cannot read the active child's title, which is why the prop had to be hardcoded. The canonical, most-performant fix is REMOVAL, not building a new title mechanism.

Purpose: Kill the dynamic-looking-but-constant API. Leave each page's own title as the single source of the visible label and the browser-tab title. Zero new client JS, zero new dependencies, zero PPR impact.
Output: `Topbar.tsx` and `layout.tsx` edited so no `pageTitle` flows; full gate green.
</objective>
