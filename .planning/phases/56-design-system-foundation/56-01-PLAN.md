<objective>
Wire dark mode so it actually works, then refine the brand color palette and establish surface elevation tokens.

Purpose: The site has `class="dark"` on `<html>` permanently, ThemeProvider sets it dynamically, but globals.css has NO `.dark {}` selector to remap base tokens. Dark mode toggle does nothing — the site always renders in light-mode colors. This plan fixes that foundation failure before any component polish begins.

Output: A complete `.dark {}` block that remaps all base tokens to their dark equivalents, refined OKLCH brand palette values, and surface elevation tokens for background/card/elevated/overlay levels.
</objective>

  <name>Task 1: Add .dark {} selector block to wire dark mode tokens</name>
  <files>src/app/globals.css</files>
  <action>
The current `globals.css` defines `--color-background-dark`, `--color-primary-dark` etc. inside `@theme {}` but there is NO `.dark {}` CSS block that reassigns the base token values when the `.dark` class is on `<html>`. This means dark mode is completely broken — all colors are permanently light.

Add a `.dark {}` block immediately after the `@theme {}` block (before the `/* Base styles */` comment). This block must reassign every base token to its dark equivalent:

```
.dark {
  --color-background: <dark value>;
  --color-foreground: <dark value>;
  --color-card: <dark value>;
  --color-card-foreground: <dark value>;
  --color-popover: <dark value>;
  --color-popover-foreground: <dark value>;
  --color-primary: <dark value>;
  --color-primary-foreground: <dark value>;
  --color-secondary: <dark value>;
  --color-secondary-foreground: <dark value>;
  --color-muted: <dark value>;
  --color-muted-foreground: <dark value>;
  --color-accent: <dark value>;
  --color-accent-foreground: <dark value>;
  --color-destructive: <dark value>;
  --color-destructive-foreground: <dark value>;
  --color-border: <dark value>;
  --color-input: <dark value>;
  --color-ring: <dark value>;
  --color-bg-overlay: <dark value>;
  --color-text-inverted: <dark value>;
  --color-text-primary: <dark value>;
  --color-text-secondary: <dark value>;
  --color-text-muted: <dark value>;
}
```

Use the existing `--color-*-dark` values from `@theme {}` as the source for each `.dark {}` property value. Map them exactly:
- `--color-background` → `oklch(0.12 0.015 260)`
- `--color-foreground` → `oklch(0.94 0.008 90)`
- `--color-card` → `oklch(0.16 0.018 260)`
- `--color-card-foreground` → `oklch(0.94 0.008 90)`
- `--color-popover` → `oklch(0.16 0.018 260)`
- `--color-popover-foreground` → `oklch(0.94 0.008 90)`
- `--color-primary` → `oklch(0.7 0.1 255)`
- `--color-primary-foreground` → `oklch(0.12 0.015 260)`
- `--color-secondary` → `oklch(0.22 0.02 260)`

- `--color-muted-foreground` → `oklch(0.85 0.015 90)` (verify: must be ≥4.5:1 on --color-card dark)

<task type="auto">
  <name>Task 2: Refine brand palette and add surface elevation tokens</name>
  <files>src/app/globals.css</files>
  <action>
With dark mode now wired, refine the OKLCH brand colors to be more distinctive than shadcn defaults, and add surface elevation tokens that downstream component polish phases will use.

**Brand palette refinement (in @theme {} light values):**

The current palette (slate blue primary, warm amber accent) is the right direction but too close to shadcn defaults in lightness and chroma. Adjust:

- `--color-primary`: Change from `oklch(0.35 0.08 255)` to `oklch(0.38 0.12 255)` — more saturated, slightly more blue. Still passes WCAG AA on white background (dark blue on light).
- `--color-accent`: Change from `oklch(0.75 0.14 65)` to `oklch(0.72 0.16 55)` — warmer hue (55° = orange-amber), slightly higher chroma for more punch as a CTA color. On white background, contrast check: `oklch(0.72 0.16 55)` is a warm amber — verify foreground `oklch(0.2 0.03 65)` is readable (should be ~5:1).
- `--color-ring`: Change from `oklch(0.45 0.1 255)` to `oklch(0.55 0.15 255)` — brighter focus ring, more visible.

**Update dark counterparts in @theme {} and in .dark {} block:**
- `--color-primary-dark`: Change from `oklch(0.7 0.1 255)` to `oklch(0.72 0.13 255)` — more saturated for dark mode visibility
- Update `.dark {}` block: `--color-primary` → `oklch(0.72 0.13 255)`

**Add surface elevation tokens to @theme {} (light values):**

After the existing `/* Base surfaces */` group, add a new `/* Surface elevation */` group:

```css
/* Surface elevation */
--color-surface-base: oklch(0.985 0.002 90);       /* = background, page canvas */
--color-surface-raised: oklch(0.995 0.001 90);     /* = card, slightly off-white */
--color-surface-elevated: oklch(1 0 0);            /* = pure white, modal/overlay */
--color-surface-overlay: oklch(0.97 0.005 255);    /* = slightly tinted for sidebar/panel */
--color-surface-sunken: oklch(0.955 0.008 90);     /* = recessed inputs, code blocks */
```

**Add dark equivalents in both @theme {} and .dark {} block:**

In `@theme {}`, add `--color-surface-*-dark` tokens:
```css
--color-surface-base-dark: oklch(0.12 0.015 260);
--color-surface-raised-dark: oklch(0.16 0.018 260);
--color-surface-elevated-dark: oklch(0.20 0.022 260);
--color-surface-overlay-dark: oklch(0.14 0.016 260);
--color-surface-sunken-dark: oklch(0.10 0.012 260);
```

In `.dark {}`, add surface remapping:
```css
--color-surface-base: oklch(0.12 0.015 260);
--color-surface-raised: oklch(0.16 0.018 260);
--color-surface-elevated: oklch(0.20 0.022 260);
--color-surface-overlay: oklch(0.14 0.016 260);
--color-surface-sunken: oklch(0.10 0.012 260);
```

**Add semantic border tokens in @theme {} (light) and .dark {} (dark):**

```css
/* In @theme {} - light */
--color-border-subtle: oklch(0.93 0.01 255);       /* very light, for dividers */
--color-border-strong: oklch(0.75 0.02 255);        /* stronger, for emphasis */

/* In @theme {} - dark variants */
--color-border-subtle-dark: oklch(0.22 0.015 260);
--color-border-strong-dark: oklch(0.38 0.025 260);

/* In .dark {} */
--color-border-subtle: oklch(0.22 0.015 260);
--color-border-strong: oklch(0.38 0.025 260);
```

Note: Do NOT add `tailwind.config.ts` — all tokens go in `@theme {}` per Tailwind v4 pattern. The DSYS-05 requirement "Tailwind config extends theme with design tokens" is satisfied by the `@theme {}` block which IS the Tailwind v4 config mechanism.
