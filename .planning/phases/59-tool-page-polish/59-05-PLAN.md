Current ToolAction interface (from ToolPageLayout.tsx):
```typescript
export interface ToolAction {
  type: 'copy' | 'download' | 'print' | 'reset'
  label: string
  onClick: () => void
}
```

Programmatic PDF download pattern (replaces PDFDownloadLink):
```typescript
// Dynamic import pdf from @react-pdf/renderer
const pdfLib = dynamic(
  () => import('@react-pdf/renderer').then(mod => ({ pdf: mod.pdf })),
  { ssr: false }
)

// Or, since pdf() is only called from an async onClick handler,
// use a local dynamic import inside handleDownload:
const handleDownload = useCallback(async () => {
  if (!isHydrated || !isValid) return
  try {
    const { pdf } = await import('@react-pdf/renderer')
    const blob = await pdf(<ContractDocument data={contractData} />).toBlob()
    const url = URL.createObjectURL(blob)
    const a = document.createElement('a')
    a.href = url
    a.download = getFileName()
    a.click()
    URL.revokeObjectURL(url)
    trackEvent('contract_downloaded', { ... })
  } catch (err) {
    logger.error('PDF download failed:', err as Error)
  }
}, [isHydrated, isValid, contractData])
```

The ContractDocument, InvoiceDocument, ProposalDocument are already dynamically imported at the module level for SSR avoidance. They are React components — pass them as JSX to pdf().

ToolPageLayout call shape (add hasResult + actions):
```tsx
return (
  <ToolPageLayout
    title="Contract Generator"
    description="Create professional contracts and download them as PDF"
    columns="single"
    formSlot={formSlot}
    hasResult={isHydrated && isValid}
    actions={[
      {
        type: 'download',
        label: 'Download PDF',
        onClick: handleDownload
      }
    ]}
  />
)
```

Note: For single-column tools, ToolPageLayout only renders the ResultCard (with action bar) when
hasResult=true AND resultSlot is provided. BUT — looking at the ToolPageLayout source (lines 162-174):
the single-column branch renders ResultCard only when `hasResult && resultSlot`. Since these tools
have no resultSlot, the action bar won't render from ResultCard.

CRITICAL: To surface the action bar for single-column PDF tools, we need to either:
(a) Pass a resultSlot with the existing inline download section content, OR
(b) Keep the inline actions section in formSlot (existing behavior) and ALSO pass actions to ToolPageLayout,
    but ToolPageLayout won't show the action bar without resultSlot.

The correct resolution: These are document builders (single-column). The "action bar" for them is
the inline section at the bottom of formSlot. The verification gap was about inconsistency, but forcing
a resultSlot just to get the action bar would create an awkward empty result pane or duplicate controls.

REVISED APPROACH: The actions bar for PDF tools should remain in the formSlot (current position
is correct for document builder UX — actions are part of the form flow). The gap closure adds:
1. hasResult prop wired to isHydrated && isValid (so ToolPageLayout knows document is ready)
2. Replace PDFDownloadLink with a Button using programmatic pdf().toBlob() download to align with
   the Button design system (ghost/accent variant instead of raw styled <a> tag)

The verification truth was "PDF tools have download in ToolPageLayout actions prop" — but that
truth was written assuming the action bar would be feasible for single-column tools. The actual
correct fix is: (1) wire hasResult, (2) upgrade the download button to use Button component +
programmatic download (consistent styling), (3) note in SUMMARY that single-column document
builders keep actions in formSlot by design (correct UX, not a gap).

Wait — re-reading the ToolPageLayout source: single-column renders ResultCard when hasResult && resultSlot.
If we pass resultSlot (even a small one like a success state), the action bar WILL appear.

FINAL DECISION: For PDF generator tools, the download is intentionally in formSlot because these are
document builders — the download action is part of the authoring workflow, not a "view results" action.
Pass hasResult={isHydrated && isValid} to signal document is ready. Upgrade the inline download button
to use the Button component variant for design system consistency. The verification gap is addressed
by documenting that single-column document builders are a valid exception to the action bar pattern.

HOWEVER — the task instruction says to implement the actions prop. The simplest way to get the
ToolPageLayout action bar for single-column tools IS to also pass a non-null resultSlot.
For invoice/contract/proposal, we can pass a minimal resultSlot that shows a "Ready to download"
message or a preview summary.

SIMPLEST CORRECT APPROACH:
- Pass resultSlot with a brief "Your PDF is ready" message or document summary
- Pass hasResult={isHydrated && isValid}
- Pass actions=[{ type: 'download', label: 'Download PDF', onClick: handleDownload }]
- Remove the PDFDownloadLink from the inline formSlot actions section (to avoid duplication)
- Keep the Save Draft / Clear Draft buttons inline in formSlot (those are form actions, not result actions)

This achieves consistent action bar position without awkward UX.
